## 1. 文档目标与适用范围

### 1.1 目标

将 v1.0 的高阶设计转化为可执行方案，解决以下问题：

## 2. MVP 范围冻结（P0）

### 2.1 MVP 核心原则

- 仅覆盖“从获客到首投”的最短闭环  
- 不在 MVP 引入复杂金融策略（如 BL 修正、行为金融修正）  
- 所有上线功能必须具备审计留痕、监控告警、回滚能力

### 2.2 首期只交付 3 条主链路

#### 链路 A：税优获客链路

`H5税优计算器 -> 注册引导 -> 渠道归因`

**必须交付**
- 税优计算器（按官方规则计算，支持分享）
- H5 渠道参数透传（utm_source / campaign_id / invite_code）
- 注册页承接与归因打点

**暂不交付**
- 社交裂变玩法（组队、排行、任务系统）
- 营销自动化工坊可视化编排

#### 链路 B：开户注册链路

`手机号注册 -> 实名/KYC -> 养老金资金账户开立`

**必须交付**
- 手机号注册与验证码限流
- 实名三要素核验 + KYC 必填项
- 单合作银行开户流程打通（先单银行，后多银行扩展）

**暂不交付**
- 会员等级体系
- 跨行归集与多账户复杂迁移

#### 链路 C：投顾首投链路

`风险测评 -> 推荐方案(3档) -> 组合一键申购 -> 持仓查看`

**必须交付**
- 标准问卷风测（C1-C5）
- TRF 五档组合（保守/稳健/平衡/积极/进取）
- 适当性拦截（产品风险等级 > 用户等级时阻断）
- 组合申购事务一致性（全成或全回滚）
- 首版持仓与收益展示

**暂不交付**
- AI 对话测评
- 事件驱动再平衡自动建议
- 复杂组合回测与场景模拟器

### 2.3 MVP 退出条件（DoD）

满足以下条件才可进入灰度：

1. 三条链路端到端主路径成功率 >= 95%（样本 n >= 200）  
2. 交易与资产核心对账日差错率 <= 0.01%（按笔数）  
3. 适当性拦截覆盖率 = 100%  
4. 审计日志完整率 = 100%（关键交易与合规事件）  
5. P1/P2 严重缺陷（阻断、资金、合规）为 0

## 4. 服务边界与核心接口（MVP）

### 4.1 服务清单

1. `user-service`：注册、实名、KYC、用户状态  
2. `pension-account-service`：养老金账户开户、状态同步  
3. `advisor-service`：风测、风险等级、组合建议  
4. `product-service`：产品主数据、风险等级、费率  
5. `trade-service`：申购下单、状态流转、回执  
6. `recon-service`：清算对账、差异工单  
7. `message-service`：交易通知、合规提醒  
8. `audit-service`：合规与交易留痕查询

### 4.2 关键接口契约（示例）

#### 风险测评提交

`POST /api/v1/advisor/risk-assessments`

请求体：

```json
{
  "userId": "u_123",
  "answers": [
    { "questionId": "q1", "value": "A" },
    { "questionId": "q2", "value": "C" }
  ],
  "source": "app"
}
```

响应体：

```json
{
  "assessmentId": "ra_001",
  "riskLevel": "C3",
  "validUntil": "2027-07-01T00:00:00Z"
}
```

错误码约定（节选）：
- `RA-4001` 问卷未完成  
- `RA-4002` 题目版本过期  
- `RA-5001` 评分引擎异常

#### 组合一键申购

`POST /api/v1/trades/portfolio-orders`

请求体：

```json
{
  "userId": "u_123",
  "planId": "plan_c3_recommend",
  "totalAmount": 12000,
  "idempotencyKey": "9f56e765-13d3-4f62-9231-8271d9f9f751"
}
```

响应体：

```json
{
  "orderId": "po_10001",
  "status": "PENDING_CONFIRM",
  "subOrders": [
    { "productId": "fund_01", "amount": 4800 },
    { "productId": "fund_02", "amount": 7200 }
  ]
}
```

错误码约定（节选）：
- `TR-4091` 重复请求（幂等命中）  
- `TR-4221` 适当性校验失败  
- `TR-5031` 通道超时（已进入补偿流程）

## 6. 关键技术策略（保证金融正确性）

### 6.1 幂等与去重

- 客户端与服务端双层幂等：`idempotency_key` + 请求摘要哈希  
- 幂等窗口：24 小时（可配置）  
- 幂等命中返回原订单，不重复下单

### 6.2 分布式事务与补偿

- 组合申购采用 Saga：`创建主单 -> 创建子单 -> 发送通道 -> 回执确认`  
- 任一子单失败触发补偿：未确认子单撤销，已确认子单进入人工差错流程  
- 订单状态机必须单向流转，禁止逆转覆盖

### 6.3 对账策略

- 对账维度：订单笔数、金额、份额、手续费、状态  
- 对账频次：日终 T+1 批处理 + 关键时段增量核对  
- 差错处理：自动生成差错工单并分级（P0资金差异/P1状态差异）

### 6.4 审计与留痕

关键事件必须写入不可篡改审计流：
- 实名与 KYC 变更
- 风测结果与适当性决策
- 下单、撤单、失败补偿
- 合规提示展示与用户确认动作

## 7. 验收口径修正（从“理想值”到“可测值”）

### 7.1 统一统计规则

- 默认统计窗口：自然日（UTC+8）  
- 默认指标口径：P95/P99、成功率、差错率均按有效请求样本统计  
- 报告需注明样本量 n、是否含重试流量、是否排除灰度用户

### 7.2 关键指标修正建议

| 指标 | v1.0 表述 | v1.1 可测口径 |
|------|-----------|---------------|
| 数据一致性 | 差异率 = 0 | 日对账差错率 <= 0.01%，且资金类差异 100% 当日建单处理 |
| 产品信息准确率 | 100% | 主数据字段准确率 >= 99.9%，关键合规字段准确率 = 100% |
| 方案生成时延 | <= 3秒 | P95 <= 3秒，P99 <= 5秒，超时需降级返回基础方案 |
| 事件触达时延 | <= 3秒 | P95 <= 3秒，P99 <= 10秒；超时记录并补发 |
| 审计日志保留 | >=20年 | 冷热分层存储策略明确，可在监管检查时按用户+时间检索 |

## 8. 测试矩阵（MVP）

### 8.1 测试类型与覆盖要求

| 测试类型 | 覆盖目标 |
|----------|----------|
| 单元测试 | 核心规则分支覆盖 >= 80% |
| 集成测试 | 跨服务核心链路 100% 覆盖 |
| E2E 测试 | 三条主链路主路径 + 关键异常路径 |
| 性能测试 | 核心 API 达到 NFR 目标，含峰值与稳态压测 |
| 安全测试 | OWASP Top 10 + 鉴权绕过 + 敏感信息泄漏 |
| 合规测试 | 适当性拦截、风险揭示、日志可追溯 |

### 8.2 高风险用例（必须自动化）

1. 重复提交同一组合申购请求，仅生成一笔有效订单  
2. 通道部分成功后失败，系统触发补偿且状态一致  
3. 风测过期用户发起新交易，交易被阻断并提示重测  
4. 用户风险等级下调后，超风险产品申购被阻断  
5. 对账发现资金差异时自动建 P0 工单并通知值班

## 9. 合规审查清单（上线门禁）

### 9.1 必审事项

| 类别 | 审查项 | 责任方 | 门禁 |
|------|--------|--------|------|
| 适当性 | 产品风险等级与用户风险等级匹配逻辑 | 合规 + 产品 + 研发 | 未通过不得上线 |
| 信息披露 | 风险提示、费率、非保本声明位置与文案 | 合规 + 法务 | 未通过不得上线 |
| 数据合规 | 最小必要采集、授权弹窗、撤回授权流程 | 合规 + 安全 | 未通过不得上线 |
| 交易留痕 | 关键交易与决策日志完整可检索 | 合规 + 研发 | 未通过不得上线 |
| AML | 可疑交易监测规则与上报流程 | 风控 + 合规 | 未通过不得上线 |

### 9.2 禁止性要求

- 禁止以“免责条款”替代适当性义务  
- 禁止在无充分披露前展示“保本/稳赚”类绝对化承诺  
- 禁止未授权采集或跨目的使用个人信息

## 10. 运行与应急（MVP）

### 10.1 灰度策略

- 灰度分层：内部白名单 -> 小流量用户 -> 扩容发布  
- 灰度期间对交易链路启用更严格告警阈值  
- 任一 P0 资金类告警触发自动暂停新增交易入口

### 10.2 应急预案最小要求

1. 交易通道不可用：切换备用通道或熔断并告知  
2. 对账异常：启动差错处理流程，冻结相关批次资金操作  
3. 数据泄露风险：按预案完成隔离、审计、上报与通知

## 13. 评审附件 A：MVP 范围图（In / Out）

本节用于评审会现场快速对齐“本期做什么、不做什么”。

### 13.1 功能范围矩阵

| 业务链路 | 功能项 | In/Out | 说明 |
|----------|--------|--------|------|
| 获客 | H5 税优计算器 | In | MVP 核心入口，必须支持分享与归因 |
| 获客 | 社交裂变任务体系 | Out | 延后到 P1/P2，避免运营系统先行复杂化 |
| 用户准入 | 手机号注册 + 验证码 | In | 必须具备限流、防刷、重试策略 |
| 用户准入 | 实名三要素 + KYC | In | 合规门槛，未完成不可交易 |
| 用户准入 | 多银行并行开户 | Out | MVP 先单银行接入，降低联调复杂度 |
| 投顾 | 标准问卷风险测评（C1-C5） | In | 作为适当性前置条件 |
| 投顾 | AI 对话式测评 | Out | 需先完成一致性校准与合规评审 |
| 投顾 | TRF 三档展示（保守/推荐/进取） | In | 用户可理解且满足首投决策需求 |
| 投顾 | TDF/BL 高级策略 | Out | 数据与解释成本较高，后续迭代 |
| 交易 | 组合一键申购（原子性） | In | 必须支持幂等和失败补偿 |
| 交易 | 产品转换 | Out | 涉及更多通道与规则，不纳入 MVP |
| 持仓 | 首版持仓与收益展示 | In | 满足“投后可见”基本体验 |
| 持仓 | 智能再平衡自动执行 | Out | 先做人工确认建议，再考虑自动化 |

### 13.2 冻结机制

- 范围冻结后，新增需求必须走“变更评审”并标注对三条主链路的影响  
- P0 迭代内新增项默认进入候选池，不直接插队  
- 任何新增需求若影响合规门禁，需同步更新第 9 章与第 15 章清单

### 14.3 接口变更治理

- 所有跨团队接口需提供：字段定义、枚举值、错误码、示例报文
