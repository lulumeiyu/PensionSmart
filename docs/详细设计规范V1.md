## 一、目的与总原则

### 1.1 目的

统一"服务级详细设计"的产出物结构与质量标准，使每份详细设计做到：

1. **可开工**——研发照单实现，不再需要口头澄清；
2. **可评审**——评审人按固定清单逐项核对；
3. **可测试**——每个设计项都有对应的验收口径；
4. **可追溯**——功能点、接口、表、错误码都有全局唯一编号并回链上游文档。

### 1.2 总原则

| 编号 | 原则 | 说明 |
|------|------|------|
| P-01 | 契约先行 | 先改 `contracts/proto` 与错误码目录，后写实现设计 |
| P-02 | 单一职责 | 设计粒度对齐《功能职责分解 V1》的功能点（D*.F*），不得合并职责 |
| P-03 | 金融正确性优先 | 涉及资金的设计必须显式回答：幂等？事务边界？补偿？对账？留痕？ |
| P-04 | 失败路径完备 | 每条主流程必须配套异常/超时/补偿路径，禁止只画 happy path |
| P-05 | 图文一致 | mermaid 图与文字描述冲突时以图为准，发现即修复 |
| P-06 | 上游可追溯 | 每个设计项标注来源（HRS v1.0 / 职责分解 / 架构 00~08 / v1.1 契约） |

### 1.3 产出物位置与命名

```
docs/detailed-design/<domain>/<service>_详细设计V<n>.md
```

- `<domain>` 取 `marketing/user/advisor/product/trade/content/data-platform/infra`
- 一个服务一份文档；跨服务的链路级设计放 `docs/detailed-design/_cross/`
- 模板：`docs/detailed-design/_TEMPLATE_详细设计.md`（新文档必须由模板复制生成）

## 二、详细设计文档结构（固定 10 章）

每份详细设计必须包含以下 10 章，**章节号与标题固定**，无内容的章节写"不适用+原因"，不得删除：

| 章 | 标题 | 必含内容 |
|----|------|----------|
| 1 | 概述与范围 | 服务定位、承载功能点（D*.F* 列表）、非目标（明确不做什么） |
| 2 | 领域模型 | 聚合/实体/值对象、不变式（invariants）、领域事件；mermaid classDiagram |
| 3 | 接口设计 | REST/gRPC/事件三类完整契约（见第三章规范） |
| 4 | 数据模型 | PG/CK 表结构 DDL、索引、加密字段、迁移脚本编号（见第四章规范） |
| 5 | 核心流程 | 每个用例一张 mermaid 时序图 + 异常路径；状态机 stateDiagram（见第五章规范） |
| 6 | 错误码与异常处理 | 本服务错误码表 + 降级/重试/补偿策略（见第六章规范） |
| 7 | 一致性设计 | 幂等/事务边界/Saga/Outbox/锁（见第七章规范）；无资金操作的服务说明理由后可简化 |

| 10 | 可观测与测试 | 指标/日志/追踪设计 + 测试矩阵（见第十章规范） |

## 三、接口设计规范

### 3.1 REST（对端，经 BFF/网关）

| 规则 | 内容 |
|------|------|
| 路径 | `/api/v1/<资源复数>`，kebab-case，如 `/api/v1/risk-assessments`；禁止动词路径（导出类操作用 `:action` 后缀，如 `/orders/{id}:cancel`） |
| 版本 | URI 版本 `v1`；破坏性变更升 `v2`，旧版本至少保留一个迭代周期 |
| 方法语义 | GET 只读幂等；POST 创建/动作；PUT 全量替换；PATCH 局部更新；DELETE 删除 |
| 请求头 | 写操作必须支持 `Idempotency-Key`；全链路必须透传 `X-Trace-Id` |
| 分页 | `?page=1&size=20`（size 上限 100），响应含 `total/page/size` |
| 时间 | 一律 RFC3339 UTC（如 `2026-07-03T02:31:00Z`），字段名 `xxxAt` |
| 金额 | 一律字符串高精度 `"12000.00"` + `currency`，禁止 JSON number 承载金额 |
| 字段命名 | JSON camelCase；枚举 UPPER_SNAKE |

**统一响应结构**（成功直接返回业务体；失败统一如下）：

```json
{
  "error": {
    "code": "TR-4221",
    "message": "适当性校验失败: 产品风险等级超出用户风险承受能力",
    "traceId": "tr_abc123",
    "details": [ { "field": "productId", "issue": "R5 > C3" } ]
  }
}
```

### 3.2 gRPC（域间同步）

| 规则 | 内容 |
|------|------|
| 契约位置 | 一律先定义于 `contracts/proto/<domain>/v<n>/`，实现方与调用方都从契约生成 |
| 命名 | Service 大驼峰名词（`Suitability`）；RPC 动词（`Check`）；message 为 `<RPC>Request/Response` |
| 字段 | 已发布字段编号禁止修改/复用；新增字段只增不改；金额用 `common.v1.Money` |
| 必带字段 | 所有 Request 含 `trace_id`；跨页查询含 `limit`（上限 500） |
| 超时/重试 | 设计文档必须给出：建议超时（默认 800ms，quant 类 3s）、是否可重试（只读可重试，写操作仅幂等接口可重试） |
| 错误 | gRPC status + `google.rpc.ErrorInfo`（携带平台错误码），语义与 REST 错误码一致 |

### 3.3 领域事件（域间异步）

| 规则 | 内容 |
|------|------|
| 信封 | 统一 `DomainEventEnvelope`（contracts/proto/common/v1），payload 为 JSON |
| 命名 | `<domain>.<PastTenseEvent>`，如 `trade.FirstPurchaseCompleted`；登记到 `EventTypes` 与本规范附录 A |
| Topic | RocketMQ：`{env}_{domain}_events`；订阅方用 consumer group `{service}-{purpose}` |
| 投递语义 | at-least-once；**消费方必须幂等**（以 `eventId` 去重） |
| 顺序 | 需要顺序的事件（如订单状态）以聚合 ID 为 sharding key |
| 演进 | payload 只增字段不删改；版本写入信封 `version` |
| 设计要求 | 详细设计必须列出：发布事件表（触发时机/payload schema）+ 订阅事件表（处理逻辑/失败策略：重试 N 次后入死信 + 告警） |

## 四、数据模型设计规范

### 4.1 PostgreSQL（OLTP）

| 规则 | 内容 |
|------|------|
| 库隔离 | 一域一库（如 `pension_trade`），**禁止跨域直连他域库**；服务内多 schema 可选 |
| 命名 | 表/字段 snake_case；表名复数（`portfolio_orders`）；不用数据库保留字 |
| 主键 | 业务主键用带前缀字符串 ID（`po_`/`u_`/`pa_`），由服务内发号（ULID/雪花），禁止依赖自增暴露规模 |
| 必备审计字段 | `created_at timestamptz not null default now()`、`updated_at`；变更敏感表加 `version int`（乐观锁） |
| 金额 | `numeric(18,2)`；比率 `numeric(10,6)`；禁止 float/double |
| 枚举 | 用 `varchar + check` 或应用层枚举，DDL 注释列出全部取值 |
| 加密字段 | 手机号/身份证/银行卡等 PII 一律密文列（`_enc` 后缀），并注明密钥来源（S2.4 HSM）；需查询的加 HMAC 索引列（`_hash`） |
| 索引 | 设计文档必须为每个查询场景列出对应索引；外键关系用应用层保证 + 索引，不建物理外键（分库分表兼容） |
| 幂等 | 幂等键唯一约束（如 `idempotency_key unique`）必须落库，不能只靠 Redis |
| 迁移 | Flyway 版本化：`V<序号>__<描述>.sql`，只前进不修改历史；DDL 全部入库到服务的 `src/main/resources/db/migration/` |

**DDL 示例（含规范要素）**：

```sql
-- V1__create_portfolio_orders.sql
CREATE TABLE portfolio_orders (
    order_id        varchar(32)  PRIMARY KEY,            -- po_ 前缀
    user_id         varchar(32)  NOT NULL,
    total_amount    numeric(18,2) NOT NULL,
    currency        varchar(3)   NOT NULL DEFAULT 'CNY',
    status          varchar(20)  NOT NULL,               -- INIT/PENDING_CONFIRM/PROCESSING/SUCCEEDED/FAILED/ROLLED_BACK
    idempotency_key varchar(64)  NOT NULL,
    version         int          NOT NULL DEFAULT 0,
    created_at      timestamptz  NOT NULL DEFAULT now(),
    updated_at      timestamptz  NOT NULL DEFAULT now(),
    CONSTRAINT uq_portfolio_orders_idem UNIQUE (idempotency_key),
    CONSTRAINT ck_portfolio_orders_status CHECK (status IN
        ('INIT','PENDING_CONFIRM','PROCESSING','SUCCEEDED','FAILED','ROLLED_BACK'))
);
CREATE INDEX idx_portfolio_orders_user_created ON portfolio_orders (user_id, created_at DESC);
```

### 4.2 ClickHouse（OLAP）

| 规则 | 内容 |
|------|------|
| 分层 | 表归属 ODS/DWD/DWS/ADS 之一，库名即层名 |
| 引擎 | 明细 MergeTree 族；`PARTITION BY` 日期、`ORDER BY` 查询主维度 |
| TTL | 明细层默认 3 年，聚合层按需；审计类冷备到对象存储（20 年） |
| 写入 | 只允许经 Kafka 管道写入，业务服务禁止直写 |

### 4.3 Redis

| 规则 | 内容 |
|------|------|
| Key 命名 | `pension:{domain}:{entity}:{id}[:{field}]`，如 `pension:trade:idem:9f56...` |
| TTL | 一律显式 TTL，禁止永久 key；幂等键 24h、会话 30min 滑动、缓存按场景 |
| 大 key | value > 100KB 或集合 > 5000 元素需拆分设计 |
| 一致性 | 缓存与 DB 的更新顺序（先 DB 后失效缓存）与容错策略必须写入设计 |

## 五、流程/状态机设计规范

| 规则 | 内容 |
|------|------|
| 用例时序图 | 每个核心用例一张 mermaid `sequenceDiagram`，参与者用真实服务名；同步调用实线、事件虚线 |
| 异常路径 | 时序图必须含 `alt/else` 覆盖：校验失败、下游超时、部分失败补偿三类 |
| 状态机 | 有状态实体必须给 mermaid `stateDiagram-v2`；**状态只能单向流转**，终态显式标注；每个迁移注明触发者与守卫条件 |
| Saga | 跨服务写操作必须画补偿路径：每个正向步骤对应的补偿动作、不可补偿步骤（如已确认子单）转人工差错的入口 |
| 定时任务 | 批处理需注明：调度周期、幂等（重跑安全）、超时告警阈值、与在线流量的隔离方式 |
| 图形校验 | 所有 mermaid 必须可渲染（提交前用 `@mermaid-js/mermaid-cli` 校验） |

## 六、错误码规范

### 6.1 格式

```
<域前缀>-<4|5><三位序号>     例: TR-4221, RA-4001, AC-4003
```

- 首位 `4` = 业务/客户端可预期错误；`5` = 系统/下游错误
- 同一错误码全局唯一语义，**登记后不得改义**，作废保留占位

### 6.2 域前缀目录

| 前缀 | 域/服务 | 前缀 | 域/服务 |
|------|---------|------|---------|
| `MK` | D1 营销获客 | `PD` | D4 产品中心 |
| `US` | D2 用户/认证 | `TR` | D5 交易 |
| `AC` | D2 养老金账户 | `RC` | D5 清算对账 |
| `RA` | D3 风测/适当性 | `CT` | D6 内容教育 |
| `AD` | D3 配置/推荐 | `DP` | S1 数据中台 |
| `GW` | S2 网关/平台 | — | — |

已登记错误码（v1.1 沿用）：`RA-4001` 问卷未完成、`RA-4002` 题目版本过期、`RA-5001` 评分引擎异常、`TR-4091` 幂等命中、`TR-4221` 适当性失败、`TR-5031` 通道超时（已入补偿）。

### 6.3 异常处理设计要求

每个错误码在设计文档中必须注明四元组：**触发条件 / 用户可见文案（合规审核）/ 系统动作（重试・降级・补偿・告警）/ 是否留痕审计**。

## 七、一致性设计规范（幂等/事务/Saga）

| 场景 | 强制方案 |
|------|----------|
| 单服务写 | PG 本地事务；事务内禁止远程调用（gRPC/HTTP/MQ 发送均不允许在事务中同步执行） |
| 写后发事件 | **Outbox 模式**：业务表与 outbox 表同事务写入，独立投递器发 RocketMQ |
| 跨服务写 | **Saga 编排**：设计文档必须给出正向步骤表 + 补偿步骤表 + 不可补偿转人工的界限 |
| 写接口幂等 | 双层：Redis `SETNX(idempotency_key)`（快速拒重）+ PG 唯一约束（最终兜底）；命中返回原结果而非报错 |
| 消费幂等 | 以 `eventId` 建消费记录表去重；重复消息静默跳过 |
| 并发控制 | 单聚合内乐观锁（`version`）；跨聚合协调用 Redis 锁（必须带看门狗续期与超时） |
| 状态机 | 状态迁移必须带前置状态条件更新（`UPDATE ... WHERE status = :expected`），影响行数为 0 视为并发冲突 |
| 重试 | 只读与幂等写可自动重试（指数退避，上限 3 次）；非幂等写禁止自动重试，转补偿或人工 |

## 八、缓存与性能设计规范

| 项 | 规范 |
|----|------|
| 默认 SLO | 核心 API P95 ≤ 500ms / P99 ≤ 1s；投顾方案 P95 ≤ 3s；交易 P99 ≤ 800ms（架构 00 §4.4） |
| 容量估算 | 设计文档必须含：预估 QPS（日常/峰值）、单请求资源画像、数据量年增长 |
| 缓存防护 | 击穿：热点 key 逻辑过期 + 单飞（singleflight）；穿透：空值缓存或布隆过滤器；雪崩：TTL 加随机抖动 |
| 本地缓存 | 读极热数据（产品要素/标签）可加 Caffeine 二级缓存，必须注明与 Redis 的失效联动 |
| 降级 | 每个含下游依赖的接口必须声明降级行为（返回缓存快照 / 静态兜底 / 显式失败），与架构 00 §4.5 对齐 |

## 九、安全与合规设计清单（逐项声明）

详细设计第 9 章必须对下表逐项写"适用（方案）/不适用（原因）"：

| 编号 | 检查项 | 依据 |
|------|--------|------|
| SEC-01 | PII 字段加密存储（`_enc`/`_hash`），密钥经 HSM | NFR-06 |
| SEC-02 | 接口鉴权与越权控制（水平越权：资源属主校验） | NFR-05 |
| SEC-03 | 写操作审计留痕（`AuditEvent` 哈希链） | v1.1 §6.4 |
| SEC-04 | 适当性前置校验（交易类）100% 覆盖 | NFR-09 |
| SEC-05 | 风险提示/免责文案经合规审核（展示类） | NFR-10 |
| SEC-06 | 个人信息最小采集 + 授权记录（PIPL） | NFR-11 |
| SEC-07 | AML 可疑交易上报数据口径（资金类） | NFR-12 |
| SEC-08 | 敏感操作二次验证（大额交易>5万短验） | D5-AC-04 |
| SEC-09 | 日志脱敏（手机号/证件号打码，禁止明文入日志） | 等保三级 |

## 十、可观测性与测试规范

### 10.1 可观测

| 项 | 规范 |
|----|------|
| 指标命名 | Prometheus：`pension_<domain>_<subject>_<unit>`，如 `pension_trade_order_total`、`pension_recon_fund_diff_total`；业务关键指标必须在设计中列表 |
| 告警 | 每个服务至少定义：可用性（错误率）、时延（P95/P99）、业务异常（如对账差异）三类告警及分级（P0/P1/P2） |
| 日志 | 结构化 JSON；关键动作固定字段：`traceId/userId(脱敏)/action/result/costMs`；等级使用规范（ERROR=需人工介入） |
| 追踪 | `X-Trace-Id` 贯穿 REST→gRPC→事件；事件信封复用 traceId |

### 10.2 测试矩阵（与 v1.1 §8 对齐）

| 类型 | 详细设计中的要求 |
|------|------------------|
| 单元测试 | 领域层核心规则分支覆盖 ≥ 80%；设计文档列出必测规则清单 |
| 集成测试 | 每个 gRPC/REST 契约至少一个正例一个反例；用 Testcontainers 起 PG/Redis |
| E2E | 标注本服务参与的主链路（v1.1 三链路）与断言点 |
| 高风险用例 | 资金/合规相关服务必须覆盖 v1.1 §8.2 的对应自动化用例（幂等重复单、部分失败补偿、风测过期阻断等） |
| 验收口径 | 引用 v1.1 §7 可测口径（统计窗口/样本量/P95 分位），禁止写"100%/0 差错"这类不可测表述（资金一致性除外） |
