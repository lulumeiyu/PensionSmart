## 一、目的与设计原则

### 1.1 目的

本文档在 HRS v1.0 的域/子域框架下，**穷举每个模块应当包含的全部功能**，并保证每个功能满足**单一职责原则（Single Responsibility Principle, SRP）**——即"每个功能只有一个被修改的理由，只做一件内聚的事"。这为后续的服务拆分、接口设计、任务分解与测试用例编写提供直接依据。

### 1.2 单一职责的判定标准

一个功能被认为符合单一职责，需同时满足：

1. **单一变更理由**：只有一类业务规则变化会导致该功能被修改。
2. **单一动作动词**：功能名称能用一个动词短语描述（如"计算""校验""渲染""同步""发布"），而非"管理并处理并通知"。
3. **无副作用越界**：功能只产出自身职责范围内的结果，跨域影响通过**发布事件**或**调用下游接口**完成，而不在本功能内直接修改他域数据。
4. **可独立测试**：给定确定输入，产出确定输出，可脱离其他功能单独验证。

### 1.3 功能表读法

每个子模块给出一张功能表，列含义如下：

| 列 | 含义 |
|----|------|
| 功能编号 | `<子域号>-Fn`，全局唯一 |
| 功能名称 | 单一动词短语 |
| 单一职责说明 | "只负责……"（正向边界）+ "不负责……"（反向边界），明确职责红线 |
| 主要输入 → 输出 | 用于验证输入/输出的确定性 |

> **反向边界（"不负责……"）是本表的关键**：它显式声明该功能*不做什么*，从而把相邻职责推给正确的功能，避免职责蔓延。

## 十、单一职责边界的横向约定

为避免跨模块的"职责撞车"，特此明确若干易混淆边界：

| 易混淆点 | 归属与边界 |
|----------|-----------|
| 适当性"判定" vs "阻断" | **判定**归 `D3.1-F5`（只给结论），**阻断**归 `D5`（交易前置控制），二者解耦 |
| 交易风控 vs 反洗钱 | **交易风控**（D5.6）关注交易行为异常；**AML**（S2.4-F4）关注资金来源与洗钱模式，职责不重叠 |
| 标签"计算" vs "查询" | 计算（D1.1-F1~F4）与查询（D1.1-F5）分离，读写职责分离 |
| 触达"频控" vs "去重" | 频控（D1.2-F6）看频次阈值；去重（D1.2-F7）看内容重复，规则来源不同 |
| 内容"创作" vs "审核" vs "推荐" | 分属 D6.3-F1 / D6.3-F2 / D6.1-F2，编辑权、合规权、算法权三权分立 |
| 消息"偏好存储" vs "免打扰过滤" | 存储（D2.4-F2）与执行过滤（D2.4-F3）分离，配置与执行解耦 |
| 数据"加密执行" vs "密钥管理" | 加解密（S2.4-F2）与密钥托管（S2.4-F3）分离，满足密钥最小暴露 |
| 事件"触发" vs "投递" | 触发（D1.2-F1~F3 产出任务）与投递（D1.2-F5 执行发送）分离 |
