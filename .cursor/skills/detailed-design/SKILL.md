# 详细设计生成技能

## 何时使用

用户要求为某个服务（如 `tax-calc-svc`、`trade-svc`）编写详细设计，或要求批量产出某域/某迭代的详细设计。

## 前置输入（开工前必须收集）

按顺序读取以下上游材料，缺一不可：

1. `docs/详细设计规范V1.md` —— 本技能的**权威规范**，所有产出以它为准
2. `docs/detailed-design/_TEMPLATE_详细设计.md` —— 复制此模板作为骨架
3. `docs/architecture/0X_<对应域>_模块设计V1.md` —— 该服务的组件定义/接口/分层/进程设计
4. `docs/个人养老金智能投顾平台_模块功能职责分解V1.md` —— 找到该服务承载的功能点（D*.F*，含"只负责/不负责"边界）
5. `docs/pension-prd-executable-v1.1.md` —— 已冻结的契约（接口/数据字典/错误码/验收口径），**沿用不推翻**
6. `contracts/proto/` 与 `services/<domain>/<service>/` —— 现有契约与脚手架（包结构、端口）

## 产出步骤

1. **定范围**：从职责分解 V1 摘出该服务全部功能点；按 v1.1 的 MVP 冻结判断哪些进本版、哪些列入"非目标"。
2. **建文档**：复制模板到 `docs/detailed-design/<domain>/<service>_详细设计V1.md`，10 章结构不得增删；空章写"不适用+原因"。
3. **领域模型**：以聚合为单位定不变式；领域事件对照规范附录 A 登记表，新事件需同步登记。
4. **接口**：REST 按规范 §3.1（幂等头/分页/金额字符串/统一错误体）；gRPC 先改 `contracts/proto` 再引用；事件给出发布表+订阅表（含失败策略）。
5. **数据**：DDL 按规范 §4（ID 前缀/审计字段/`_enc`+`_hash` 加密列/唯一幂等约束/索引对应查询场景/Flyway 编号）；Redis key 按 `pension:{domain}:{entity}:{id}` + 显式 TTL。
6. **流程**：每个用例一张 `sequenceDiagram`，**必须**含 alt/else 覆盖校验失败、下游超时、部分失败补偿；有状态实体给单向 `stateDiagram-v2`。
7. **错误码**：按 `<域前缀>-<4|5xxx>` 分配（前缀目录见规范 §6.2），每码给四元组：触发条件/合规文案/系统动作/是否留痕；不与已登记码冲突。
8. **一致性**：回答四问——幂等键是什么？事务边界在哪（事务内禁远程调用）？发什么 Outbox 事件？跨服务写的 Saga 正向/补偿步骤表？
9. **安全合规**：SEC-01~09 逐项声明适用/不适用。
10. **可观测与测试**：指标按 `pension_<domain>_<subject>_<unit>` 命名；测试矩阵对齐 v1.1 §8（资金类必须覆盖 §8.2 高风险用例）。

## 质量门禁（提交前自查）

- 用 `@mermaid-js/mermaid-cli`（`npx -y -p @mermaid-js/mermaid-cli mmdc`）逐图校验可渲染
- 对照模板底部 DoD 清单逐项勾选
- 金额出现处检查：字符串高精度，无 JSON number / float
- 错误码/事件类型与规范附录 A、§6.2 无冲突；新增的要在规范附录 A 登记
- 验收表述可测（P95/样本量/统计窗口），禁止"100%、0 差错"类不可测表述（资金一致性除外）

## 常见错误（避免）

- 只画 happy path，缺补偿路径 → 违反规范 P-04
- 在详细设计里"顺手"扩功能点范围 → 违反 P-02，超出部分放"非目标"或另立功能点评审
- gRPC 契约只写在文档里不改 `contracts/proto` → 违反 P-01
- 把适当性"阻断"设计进 D3、或把"判定"设计进 D5 → 违反职责分解 V1 §十 的边界约定
